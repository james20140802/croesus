# Status Dashboard

`python -m croesus.jobs.local_sync --status` prints a health snapshot of the
local Croesus pipeline without running any jobs. It is safe to run at any time.

## What it shows

### Data freshness

One row per data domain showing the last successful run timestamp, the date of
the most recent data, and whether the domain is fresh or stale:

```
Data freshness:
  prices               fresh    data=2026-06-12  (last success 2h ago)
  fx                   stale    data=2026-06-10  (last success exceeds 48h threshold)
  macro_daily          fresh    data=2026-06-12  (last success 1h ago)
  ...
```

### Latest reports

The most recent registered file path per report type.  Every report write is
recorded in the `reports` DB table (Sprint 012), so this section needs no
filesystem scan:

```
Latest reports:
  macro                as_of=2026-06-12  reports/macro/2026-06-12/macro.md
  performance          as_of=2026-06-11  reports/performance/2026-06-11/performance.md
  portfolio_action     as_of=2026-06-11  reports/portfolio_action/2026-06-11/portfolio_action.md
  screening            as_of=2026-06-12  reports/screening/2026-06-12/<run-id>.md
```

If no reports have ever been registered the section shows `(none registered)`.

### Data quality

Count of ERROR-severity `data_quality_issues` rows created in the last 48 hours:

```
Data quality: 2 error(s) in last 48h
```

Errors indicate that a computation used a fallback value (missing price, missing
FX rate, etc.) and the results in the related report may be misstated.

### Pending approvals

Count of trade proposals that are in `pending` state and have not yet expired:

```
Approvals: 1 pending
```

Pending proposals expire after 7 days. Use
`python -m croesus.jobs.list_pending_approvals` to see details.

### Overall verdict

```
Overall: READY
```

| Verdict    | Condition                                               |
|------------|---------------------------------------------------------|
| `DEGRADED` | Any data-quality ERROR in the last 48 hours.            |
| `STALE`    | No errors, but at least one freshness domain is due.   |
| `READY`    | No errors and all domains are fresh.                    |

`DEGRADED` takes precedence over `STALE`. A `STALE` system is safe to read
but needs a sync run. A `DEGRADED` system has at least one value computed from
a fallback â€” inspect the relevant report's data-quality block before acting on
the numbers.

## The `reports` DB table

Schema (see `croesus/db/schema.sql`, Sprint 012 section):

| Column        | Type      | Notes                                                     |
|---------------|-----------|-----------------------------------------------------------|
| `report_id`   | TEXT PK   | UUID hex, auto-generated by `register_report`.            |
| `report_type` | TEXT      | Stable contract: `macro`, `screening`, `portfolio_action`, `performance`. |
| `as_of_date`  | DATE      | The data date the report covers (nullable).               |
| `path`        | TEXT      | Absolute or relative path to the file on disk.            |
| `format`      | TEXT      | `markdown` or `csv`, inferred from the suffix.            |
| `run_id`      | TEXT      | Links to the rebalance or screening run (nullable).       |
| `created_at`  | TIMESTAMP | `now()` at insert time.                                   |

`latest_reports()` in `croesus/reports/registry.py` returns the newest row per
`report_type` using a window function (max `created_at`, tie-break max
`report_id`).

## Adding a new report type

1. Add a `REPORT_TYPE_<X> = "<x>"` constant in `croesus/reports/registry.py`.
2. Call `register_report` or `register_many` immediately after writing the file,
   passing the connection that is already open in the writer or the job.
3. The new type appears automatically in `--status` and in `latest_reports()`.
